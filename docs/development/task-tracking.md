# 任务追踪与 Agent 协作规范

## 文件职责
- `AGENTS.md`：接手入口、开发边界和检查清单。
- `TASKS.md`：当前任务摘要，不记录详细流水。
- `dev-log/`：每项重要任务的背景、决策、实现、验证和部署记录。
- `docs/`：长期有效的架构、接口、开发和部署事实。

## Task ID 与状态
使用 `GOV-*`、`FRONTEND-*`、`API-*`、`WORKFLOW-*`、`RUNTIME-*`、`STORAGE-*`、`INFRA-*`、`TEST-*`、`DOCS-*`，编号在模块内递增且不得重用。

- `Backlog`：已识别，尚未开始。
- `In Progress`：正在审计、实现、验证、部署或等待评审。
- `Blocked`：被权限、外部系统、依赖或决策阻塞。
- `Done`：实现和真实验证完成，必要提交和部署已完成，日志已更新。

负责人明确禁止 Push 或任务不需要部署时，必须在日志中记录例外。未提交、未验证或未记录的任务不能标记 Done。

## 工作流程
1. 阅读 `AGENTS.md`、`TASKS.md` 和相关文档。
2. 在真实环境检查 Git 状态并保护已有修改。
3. 创建或更新 Task ID 和 `dev-log/YYYY-MM-DD-task-name.md`，设为 `In Progress`。
4. 重要决策、范围变化和新问题立即写入日志与看板。
5. 只修改当前任务文件，不回退无关内容。
6. 当前远端必须在 SSH 登录后手工修改，禁止从本地同步项目文件、代码或补丁。
7. 正式工作流配置以 Supabase 为唯一事实源。

## 验证与完成
按风险执行并如实记录：`npm run lint`、`npx tsc --noEmit --pretty false`、`npm run build`、服务状态、本机接口、公网页面和真实工作流。未执行写 `未执行` 或 `pending`。

Done 前确认实现、验证、必要重启、Commit/部署/Push 状态、看板和日志均准确，新风险已进入 Backlog 或 Blocked。新 Agent 不得把本地 checkout、历史 JSON 或聊天记忆当作远端事实。

## 双环境一致性交付

1. 开始前分别审计本地与远端真实代码和 Git 状态。
2. 本地与远端分别手工实现同一目标业务语义，禁止通过文件、补丁或 Git 同步。
3. 分别执行适用检查；远端额外验证服务、接口和公网页面。
4. 开发日志分别记录两端修改范围、验证结果、差异和未完成项。
5. 任一环境未完成时，不得标记 Done。
- 远端修改验证通过后，在远端及时提交并 `git push`；任务记录提交号和推送结果。
- AdsPower 页面状态或自动化日志不足以定位时，通过治理规范指定的 VNC 地址观察执行现场。
