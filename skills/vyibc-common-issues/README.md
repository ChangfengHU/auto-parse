# 常见问题与解决方案 (Common Issues)

本页记录了 `doouyin` 项目中遇到的典型技术障碍及其解决方法。

---

## 1. 浏览器启动失败：Permission Denied (13)

### 现象
在自动化执行工作流时，后台日志报错：
`bind() failed: Permission denied (13)` 或 `Playwright connection timeout`。
浏览器始终卡在 `about:blank` 或完全无法打开。

### 原因
- **特权端口限制**：macOS/Linux 系统默认规定 1024 以下的端口（如原配置的 `1009`）为特权端口，普通用户无权绑定。
- **端口被占用**：之前的残留进程未释放端口。

### 修复方案
将 Chrome 的远程调试端口修改为 1024 以上的非特权端口（建议使用 `10009`）。
**修改路径**：`lib/persistent-browser.ts`
```typescript
'--remote-debugging-port=10009',
```

## 2. 浏览器启动卡死：孤儿进程与 SingletonLock

### 现象
浏览器配置正确，但尝试启动时由于“已有一个实例在运行”或“无法获取锁”而超时。

### 原因
- **进程残留**：浏览器非正常关闭（如程序崩溃、强制停止）导致旧的 Chrome 进程变成了“孤儿进程”，依然持有数据目录的控制权。
- **锁文件未删除**：Chrome 在数据目录中生成的 `SingletonLock` 文件未被正常清理，阻止了新实例启动。

### 修复方案
在 `getPersistentContext` 启动前增加自愈逻辑：
1. **清理进程**：扫描并结束所有占用该 `user-data-dir` 的进程。
2. **强制解锁**：自动删除数据目录下的 `SingletonLock` 文件。

---

## 3. 工作流执行：导航节点死锁/卡顿

### 现象
工作流第一个“导航”节点执行后，浏览器打开但页面不刷新，或引擎显示正在执行但无反馈。

### 原因
- **冗余跳转**：引擎层面的前置自动导航与导航节点内部的 `page.goto` 发生了冲突。
- **Next.js 服务死锁**：在开发环境下并行处理 API 请求与浏览器请求时可能产生锁竞争。

### 修复方案
1. **智能跳转**：在 `navigate.ts` 中增加检查，若当前页面已在目标 URL，则跳过 `goto` 仅等待加载。
2. **逻辑解耦**：修改 `engine.ts`，对于 `navigate` 类型节点，跳过引擎的前置导航逻辑，由节点完全自主控制。
